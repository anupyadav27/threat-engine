# Onboarding Engine & Scan Orchestration

> Central hub for cloud account management and pipeline orchestration.
> Last updated: 2026-03-17

---

## Overview

The onboarding engine manages cloud account registration and serves as the orchestration hub for the entire scan pipeline. It owns the `scan_orchestration` table — the single source of truth linking all engine scan IDs.

- **Port**: 8010
- **Database**: `threat_engine_onboarding`
- **Image**: `yadavanup84/threat-engine-onboarding-api:v-full-data`
- **K8s manifest**: `deployment/aws/eks/engines/engine-onboarding.yaml`

### Relationships

```
scan_orchestration
├── orchestration_id ──→ all engines reference this as their pipeline context
├── tenant_id ─────────→ tenants.tenant_id (onboarding DB)
├── execution_id ──────→ executions.execution_id (scheduler)
├── schedule_id ───────→ schedules.schedule_id
├── discovery_scan_id ─→ discoveries DB: discovery_report.discovery_scan_id
├── check_scan_id ─────→ check DB: check_findings.check_scan_id
├── inventory_scan_id ─→ inventory DB: inventory_findings.inventory_scan_id
├── threat_scan_id ────→ threat DB: threat_findings.threat_scan_id
├── compliance_scan_id → compliance DB: compliance_report.compliance_scan_id
├── iam_scan_id ───────→ iam DB: iam_findings.iam_scan_id
└── datasec_scan_id ───→ datasec DB: datasec_findings.datasec_scan_id
```

**Note**: These are NOT foreign keys — they're string references across separate databases on the same RDS instance.

## orchestration_id vs scan_run_id

- `orchestration_id` = `scan_run_id` — they are the **same value**
- Generated once by onboarding: `str(uuid.uuid4())`
- Passed to all engines as their pipeline context
- Each engine generates its OWN `scan_id` internally and writes it back

```
orchestration_id: 337a7425-5a53-4664-8569-04c1f0d6abf0  (pipeline-wide)
  ├── discovery_scan_id:  disc_abc123...     (generated by discovery engine)
  ├── check_scan_id:      chk_def456...      (generated by check engine)
  ├── inventory_scan_id:  inv_ghi789...      (generated by inventory engine)
  ├── threat_scan_id:     threat_bfed9ebc... (generated by threat engine)
  ├── compliance_scan_id: comp_jkl012...     (generated by compliance engine)
  ├── iam_scan_id:        iam_mno345...      (generated by IAM engine)
  └── datasec_scan_id:    ds_pqr678...       (generated by datasec engine)
```

## Pipeline Execution Flow

### 3 Trigger Paths

**A. Manual Scan** (via API):
```
POST /api/v1/cloud-accounts/{account_id}/scan
  → create_orchestration_record()
  → publish scan_requested to SQS (if SQS_PIPELINE_QUEUE_URL set)
  → OR call engines directly via engine_client.py
```

**B. Scheduled Scan** (via cron):
```
scheduler_service.py poll loop
  → get_due_schedules()
  → TaskExecutor.execute_scan()
  → create_orchestration_record()
  → publish scan_requested to SQS
```

**C. SQS Pipeline Worker** (automated orchestration):
```
shared/pipeline_worker/worker.py
  → polls SQS_PIPELINE_QUEUE_URL
  → receives scan_requested event
  → runs run_pipeline(event, sqs)
```

### Pipeline Stage Order

```
Stage 1: Discovery (parallel or single-pod)
  POST /api/v1/discovery/parallel {orchestration_id}
  → Waits for completion
  → Writes discovery_scan_id to scan_orchestration

Stage 2: Inventory (can run parallel with Check)
  POST /api/v1/inventory/scan/discovery {tenant_id, orchestration_id}
  → Reads discovery_findings
  → Writes inventory_scan_id

Stage 3: Check
  POST /api/v1/scan {orchestration_id}
  → Reads discovery_findings via discovery_scan_id from orchestration
  → Writes check_scan_id

Stage 4: Threat
  POST /api/v1/scan {orchestration_id, scan_run_id, tenant_id, cloud}
  → Reads check_findings + discovery_findings
  → Writes threat_scan_id

Stage 5 (parallel): Compliance + IAM + DataSec
  POST /compliance/api/v1/scan {orchestration_id}
  POST /iam/api/v1/scan {orchestration_id, csp}
  POST /datasec/api/v1/scan {orchestration_id, csp}
  → All three read from check_findings + threat_findings
  → Write compliance_scan_id, iam_scan_id, datasec_scan_id
```

### Event Publishing (SQS)

If `SQS_EVENTS_QUEUE_URL` is configured, the pipeline worker publishes:
- `stage_complete(orchestration_id, tenant_id, account_id, provider, stage_name, scan_id)`
- `stage_failed(orchestration_id, tenant_id, account_id, provider, stage_name, error_msg)`
- `scan_complete(orchestration_id, tenant_id, account_id, provider)` — after all stages

## How Engines Read Orchestration

All engines use `shared/common/orchestration.py`:

```python
# Step 1: Receive orchestration_id in POST body
# Step 2: Hydrate full context
ctx = get_orchestration_metadata(orchestration_id)
# ctx = {tenant_id, account_id, provider, credential_ref,
#        include_services, discovery_scan_id, check_scan_id, ...}

# Step 3: Execute scan using hydrated context
# Step 4: Write back own scan_id
update_orchestration_scan_id(orchestration_id, "threat", threat_scan_id)
```

This pattern keeps engines loosely coupled — they only need `orchestration_id`, not full parameters.

## Scan Status API

```
GET /api/v1/scans/recent?tenant_id=X&limit=10
→ Ordered by started_at DESC from scan_orchestration

GET /api/v1/scans/{scan_run_id}/pipeline?tenant_id=X
→ Per-stage status derived from engines_requested vs engines_completed:
   - engine in engines_completed → "completed"
   - overall_status == "running" and first incomplete → "running"
   - overall_status in ("failed","error") and first incomplete → "failed"
   - else → "pending"
```

## Inter-Engine Communication: DB Polling vs SQS

**Current approach**: Database polling (engines read `scan_orchestration` to find upstream scan IDs)

**Why this is sufficient**:
1. Pipeline is sequential — each stage waits for the previous one
2. `scan_orchestration` is the single source of truth
3. SQS is only used for pipeline-level orchestration (scan_requested → pipeline worker)
4. No real-time messaging needed between engines
5. Each engine's API call is synchronous — pipeline worker waits for HTTP response

**Where SQS IS used**:
- `SQS_PIPELINE_QUEUE_URL`: Onboarding → Pipeline Worker (trigger scan)
- `SQS_EVENTS_QUEUE_URL`: Pipeline Worker → external consumers (monitoring)
- `SQS_DISCOVERY_QUEUE_URL`: Dispatcher → KEDA workers (parallel discovery)

**Adding SQS between engines would only help if**:
- Engines needed async notification of upstream completion
- Multiple consumers needed to react to same event
- Neither applies today — DB polling is the right choice
